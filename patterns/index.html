The 28-Pattern Matrix

A 7×6 framework for agent architecture.

The model spends. The harness budgets. This catalog is the vocabulary you can put into your project tomorrow.

Most "agent architecture" guides give you a flat list — Reflection, ReAct, Multi-Agent, Tree of Thoughts, Reflexive Metacognitive, and so on. A flat list answers what patterns exist. It does not answer where my problem sits, and which pattern lives at that coordinate.

A loan-evaluation agent crashes not because Reflection is missing, but because Perception-stage budget allocation dropped the disqualifying document. A multi-agent code reviewer drifts not because ReAct is wrong, but because Collaboration topology without conflict resolution allows contradictory edits to accumulate. Knowing which axis the failure sits on matters more than knowing the failing pattern's name.

Two axes

Cognitive function (the rows) — what kind of work the agent is doing. Seven categories: Perception, Memory, Reasoning, Action, Reflection, Collaboration, Governance.

Execution topology (the columns) — what shape the work takes. Six archetypes: Chain, Route, Parallel, Loop, Hierarchy, Orchestrate.

Crossed, the axes form a 7×6 = 42-cell matrix. Twenty-eight cells are populated by named patterns — some borrowed from existing literature (RAG, Chain-of-Thought, Plan-and-Execute), some named here for cells that had no established name (Context Triage, Failure Journal, Observability Harness, others). Fourteen cells are intentionally empty — they mark industrial gaps and open-research frontiers.

A note on Choreography. Choreography is sometimes listed as a topology. We treat it instead as a coordination mode that operates across topologies — closer in spirit to the Teams pattern than to an independent execution shape — so it is described in the patterns text rather than given its own column.

Pattern white papers · 28 core + 5 extension = 33

Every named cell has a one-page enterprise white paper: what problem it solves, what coordinate it sits at, where it tends to break, and what to monitor once it ships. Written for the whole organization — product, architecture, R&D, compliance, management — not just engineers.

Five extension patterns (M5 Procedural Memory · R5 Talker-Reasoner · A4 Minimal Tool Set · C5 Sub-Agent Isolation · G4 Hooks Pipeline) sit alongside the 28 core patterns; each has its own white paper, bringing the total to 33.

Following the ADPS Chinese-first publishing pipeline, the 33 white papers are currently published in Chinese. English editions are in progress; pattern links below point to the Chinese white papers in the meantime.

Perception · 4 core

  • P1 · Context Triage — When candidate information exceeds the context window budget, decide who enters, who waits at the door, and who is not pre-loaded at all.
  • P2 · Semantic Compaction — When the agent's context fills up, what to keep and how aggressively to compress determines whether it can still reason clearly afterward.
  • P3 · Progressive Discovery — Facing an unfamiliar information space, the agent scans broadly first, then focuses, then drills — a three-phase loop until the space is mapped.
  • P4 · Multi-Modal Fusion — Inputs arrive as images, text, tables, logs. Convert each to its LLM-friendliest form, then merge and pass to reasoning.

Memory · 4 core + 1 extension

  • M1 · Hierarchical Retention — Slice agent memory into layers by scope; each layer has its own storage, lifecycle, and load strategy. New sessions load coarse-to-fine.
  • M2 · RAG Pipeline — Hand the agent an external library it can query, filter, and iterate against whenever the needed knowledge exceeds context capacity.
  • M3 · Progress Tracking — On long tasks, the agent maintains an explicit structured todo list, externalizing state each round so both itself and the user can see progress.
  • M4 · Failure Journals — Each failure recorded as a structured entry (which step, what error, root cause, fix) — persisted across tasks, recalled on similar future runs.
  • M5 · Procedural Memory extension — Solidify successful execution flows for a task class into named, loadable, reusable assets; next time the same class arrives, skip exploration and run the main path.

Reasoning · 4 core + 1 extension

  • R1 · Chain-of-Thought — Make the model's reasoning explicit as a structured, storable, auditable, cross-model-portable record. "Why did it conclude that" becomes inspectable.
  • R2 · Complexity-Based Routing — Before entering the main loop, route by complexity signal to model + effort tier; cheap requests run on cheap models, hard ones on expensive ones. Sort by triage, not by uniform retry.
  • R3 · Parallel Exploration — Deliberately fire N independent reasoning chains for the same query, then aggregate into a single answer. Compute traded for accuracy.
  • R4 · Iterative Hypothesis Testing — Form a hypothesis, validate with evidence, revise on the result; loop until evidence converges or the iteration cap is reached. Scientific method, embedded into reasoning.
  • R5 · Talker-Reasoner extension — Split the agent into a fast shallow Talker and a slow deep Reasoner, coordinated through a shared belief state. The agent can think deeply and still reply to the user in real time.

Action · 4 core + 1 extension

  • A1 · Tool Dispatch — Before each action step, the engineering layer picks the most appropriate tool from the registry using metadata — not by letting the model decide on the fly.
  • A2 · Plan-and-Execute — Agent generates a complete plan first (with dependencies, resource estimates, approval nodes), then executes it; on deviation, replan locally rather than rewrite globally.
  • A3 · Prompt Chaining — Decompose a complex task into a series of smaller tasks, each driven by its own prompt; the output of one step feeds the next.
  • A4 · Minimal Tool Set extension — Cap the agent's tool set at a reasonable ceiling (usually 10–30): drop low-frequency tools, merge similar ones, push secondary tools into sub-agents. Give the model room to choose.
  • A5 · Guardrail Sandwich — Around every side-effecting or high-risk action, wrap a pre-check / action / post-check sandwich. Move the agent from "free action" to "supervised action".

Reflection · 4 core

  • F1 · Generator-Critic — Two roles inside one agent: Generator writes, Critic reviews; loop until the critic passes or the iteration cap is reached.
  • F2 · Skill Package — Package repeatedly successful workflows into named, loadable, reusable assets; on the next similar task, load instead of rediscovering from scratch.
  • F3 · Experience Replay — When a new task arrives, the agent actively retrieves similar historical task trajectories and injects reusable parts into current context. What was learned does not go to waste.
  • F4 · Self-Heal Loop — On a deterministic failure signal, the agent auto-diagnoses, fixes, validates, loops — until convergence or a circuit breaker.

Collaboration · 4 core + 1 extension

  • C1 · Hierarchical Delegation — A supervisor agent dynamically splits a task, dispatches to N worker agents, and merges results. Classic multi-agent supervisor/worker coordination.
  • C2 · Fan-Out / Gather — An orchestrator dispatches independent sub-tasks to N sub-agents in parallel; an aggregator merges them via a defined strategy. N× compute for 1/N latency.
  • C3 · Adversarial Review — Pair a generator agent with one or more independent critics; critics must use different models, different families, different prompting incentives. Structured adversarial loop improves the output.
  • C4 · Handoff Chain — Split a long flow into N agents with clear responsibilities; each hands off a structured HandoffPacket (not raw text) to the next. Each agent is good at one thing.
  • C5 · Sub-Agent Isolation extension — Sub-agents run in isolated context; before returning, they must compress into a schema-shaped artifact. The parent agent reads the artifact, not the raw trajectory.

Governance · 4 core + 1 extension

  • G1 · Approval Gate — Before any high-blast-radius or irreversible action, route by risk to one of three lanes: auto-pass / log / human review. Humans retain final veto at the critical decision points.
  • G2 · Blast Radius Control — Classify actions (readonly / mutating / catastrophic) and use nested engineering boundaries to lock the maximum loss from a single failure.
  • G3 · Progressive Commitment — The agent starts read-only; permissions unlock in stages — to writes, then to irreversible actions. Each tier earned with evidence; failures trigger immediate demotion.
  • G4 · Hooks Pipeline extension — Attach deterministic interception layers before and after tool calls and agent steps. Strip from the prompt anything the model shouldn't be left to decide — guarantee it in hooks.
  • G5 · Observability Harness — Attach five-dimensional telemetry across the agent lifecycle; thread traces across agents. Replay after the fact, audit for compliance, decide for product.

The methodology

The matrix becomes operational through the Pattern Selection Card — a three-panel decision tool:

  • ASSESS — what cognitive functions does the task need?
  • ROUTE — what topology suits the constraints?
  • SELECT — which named patterns sit at those intersections?

The card turns "I'm building an agent for X" into a candidate pattern list in under fifteen minutes.
